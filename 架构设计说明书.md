## 一、项目概述

### 1.1 项目背景

高校后勤服务（维修、保洁、搬家、打印、洗衣等）长期存在以下痛点：

| 痛点 | 现状 | 影响 |
|------|------|------|
| 流程线下化 | 学生电话/微信预约，宿管手工记单 | 效率低、易遗漏、无记录 |
| 信息不对称 | 学生不知道有哪些服务、价格多少 | 体验差、投诉多 |
| 派单靠人工 | 管理员手动分配任务给师傅 | 不透明、响应慢 |
| 营销手段弱 | 优惠券靠纸质发放或群消息 | 触达率低、核销难统计 |
| 数据无沉淀 | 订单数据散落在聊天记录里 | 无法分析、无法优化 |

2024年高校后勤数字化政策加速推进，亟需一套能支撑单校日均10万+订单的线上化平台。

### 1.2 项目目标

构建一套 **多租户SaaS高校智慧后勤平台**，实现：

- **全流程线上化**：从服务浏览 → 下单 → 支付 → 派单 → 服务 → 评价，完整闭环
- **高并发可支撑**：秒杀抢券TPS 1万+，日均订单处理10万+
- **数据驱动运营**：订单分析、用户画像、优惠券ROI统计
- **多校复用**：一套系统服务多所高校，数据隔离、独立配置

### 1.3 项目名称释义

**校享云 (ShareCampus Cloud)**：
- **校** — 聚焦高校场景
- **享** — 分享经济理念，师生共享校园生活服务；享受便捷体验
- **云** — SaaS云平台，多租户、弹性扩展

## 三、技术选型

### 3.1 选型原则

1. **生产验证优先** — 选择有大厂大规模生产实践的组件
2. **Spring生态整合** — 与Spring Boot/Cloud无缝集成
3. **可运维性** — 管理平台完善、社区活跃、文档齐全
4. **渐进式复杂度** — 先用单体验证业务，再拆微服务，再引入中间件

### 3.2 技术栈清单

#### 3.2.1 基础框架

| 技术 | 版本 | 用途 | 选型理由 |
|------|------|------|----------|
| Java | 17 LTS | 开发语言 | 长期支持、虚拟线程、模式匹配 |
| Spring Boot | 3.2.x | 应用框架 | 最新稳定版，原生镜像支持 |

| Spring Cloud Gateway | 匹配SCA | API网关 | 响应式非阻塞，性能优于Zuul |

### 3.3 为什么不选其他方案

| 候选方案 | 为什么不选 |
|----------|-----------|
| MyCat | 代理层架构，需额外部署运维；ShardingSphere-JDBC是SDK级，侵入小 |
| Dubbo | 功能强大但生态偏RPC；Spring Cloud + OpenFeign RESTful更通用 |
| Canal + MQ缓存同步 | 本项目MySQL→ES同步采用Canal，属于数据同步而非缓存同步，是合理的binlog订阅场景 |
| Elastic-Job | 已停止维护；XXL-JOB社区更活跃 |
| Kotlin | 团队Java栈更熟练，Java 17已补齐大量语法糖 |

### 4.2 认证授权服务 (auth-service)

**功能清单**：
- [x] 多端登录（学生小程序/Web/App，师傅端，运营端）

### 5.1 订单状态机设计 ★★★★★

**设计模式**：状态模式 (State Pattern)

```
                    ┌──────────────┐
                    │ PENDING_PAY  │ ←── 订单创建
                    │   (待支付)    │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │ 支付成功    │            │ 超时/主动取消
              ▼            │            ▼
    ┌─────────────────┐    │    ┌──────────────┐
    │ PENDING_ACCEPT  │    │    │  CANCELED    │
    │   (待接单)       │    │    │   (已取消)    │
    └────────┬────────┘    │    └──────────────┘
             │             │
    ┌────────┼────────┐    │
    │ 师傅接单 │        │ 超时无人接单
    ▼        ▼        ▼
┌────────┐ ┌────────┐
│IN_PRO… │ │CANCELED│
│服务中   │ │(退款)  │
└───┬────┘ └────────┘
    │
    │ 服务完成
    ▼
┌──────────┐
│COMPLETED │ ←── 完成后可评价
│ (已完成)  │
└──────────┘
```

**代码结构**：

```java
// 状态接口
public interface OrderState {
    OrderStatus getStatus();           // 当前状态枚举
    void pay(OrderContext ctx);        // 支付
    void accept(OrderContext ctx);     // 接单
    void startService(OrderContext ctx); // 开始服务
    void complete(OrderContext ctx);   // 完成
    void cancel(OrderContext ctx, String reason);  // 取消
    void applyRefund(OrderContext ctx, String reason); // 申请退款
}

// 具体状态实现
PendingPayState       → 待支付状态
PendingAcceptState    → 待接单状态
InProgressState       → 服务中状态
CompletedState        → 已完成状态
CanceledState         → 已取消状态
RefundingState        → 退款中状态
RefundedState         → 已退款状态

// 状态上下文
public class OrderContext {
    private Order order;
    private OrderState currentState;
    
    public void setState(OrderState state) {
        this.currentState = state;
        this.order.setStatus(state.getStatus());
    }
}
```

**状态转换约束表**：

| 当前状态 | 允许的操作 |
|----------|-----------|
| PENDING_PAY | pay(), cancel() |
| PENDING_ACCEPT | accept(), cancel() |
| IN_PROGRESS | startService(), complete(), applyRefund() |
| COMPLETED | (终态，仅可评价) |
| CANCELED | (终态) |

### 5.2 高并发抢券设计 ★★★★★

```

                            │
                            ▼
                  ┌──────────────────┐
                  │  Sentinel 限流    │  ←── 入口限流 (QPS限制)
                  │  (gateway层)      │

                  │  Redis + Lua     │  ←── 原子扣库存
                  │  脚本执行         │     库存key: coupon:stock:{templateId}
                  │  EVALSHA         │     已抢key: coupon:grabbed:{templateId}
                  └────────┬─────────┘
                           │
              ┌────────────┼────────────┐
              │ 库存>0      │            │ 库存=0
              ▼            │            ▼
        ┌──────────┐       │      ┌──────────┐
        │ 扣减成功  │       │      │ 返回失败  │
        │ 发送MQ   │       │      │ "已抢光"  │
        └────┬─────┘       │      └──────────┘
             │             │
             ▼             │
    ┌────────────────┐     │
    │ RabbitMQ       │     │
    │ coupon.grab    │     │
    │ 队列异步落库    │     │
    └────────┬───────┘     │
             │             │
             ▼             │
    ┌────────────────┐     │
    │ MySQL持久化     │     │
    │ 扣库存+发券     │     │
    └────────────────┘     │
```

**Lua脚本（原子扣库存 + 去重）**：

```lua
-- KEYS[1]: 库存key    coupon:stock:{templateId}
-- KEYS[2]: 已抢key    coupon:grabbed:{templateId}
-- ARGV[1]: 用户ID
-- ARGV[2]: 扣减数量(默认1)

-- 检查是否已抢过
local grabbed = redis.call('sismember', KEYS[2], ARGV[1])
if grabbed == 1 then
    return -2  -- 已抢过，不能重复抢
end

-- 检查库存
local stock = tonumber(redis.call('get', KEYS[1]) or '0')
if stock <= 0 then
    return -1  -- 库存不足
end

-- 扣库存 + 标记已抢
redis.call('decrby', KEYS[1], tonumber(ARGV[2]))
redis.call('sadd', KEYS[2], ARGV[1])

return 1  -- 抢券成功
```

**异步落库流程**：
1. Lua脚本成功 → 发送MQ消息 `coupon.grab.success`
2. 消费者接收 → 开启Seata分布式事务
3. 扣减MySQL库存 → 写入用户优惠券记录
4. 事务提交 → 最终一致性保证
5. 失败 → 重试3次 → 人工补偿

### 5.6 结算分账设计 ★★★★

**场景**：一笔保洁订单59元，学生用了一张10元券实付49元。按规则平台抽佣15%，师傅到手多少？

```
分账计算:
  订单实付: ¥49.00（优惠券10元已由平台承担）
  平台抽佣: ¥49.00 × 15% = ¥7.35
  师傅到手: ¥49.00 - ¥7.35 = ¥41.65
  
  T+1 结算:
  师傅钱包待结算+¥41.65 → 次日自动转入可提现余额
  
  师傅提现:
  可提现¥41.65 → 申请提现 → 冻结 → 微信企业付款 → 已提现
```

**结算状态机**：
```
订单支付成功 → WAIT_SETTLE(待结算)
              → T+1定时任务扫描 → SETTLED(已结算，转入可提现)
              → 师傅申请提现 → 可提现减，冻结加 → WITHDRAW_PENDING(待审核)
              → 运营审核/自动通过 → WITHDRAW_SUCCESS(已提现)
```

**钱款流转保障**：
```java
// 乐观锁防并发

SET available_balance = available_balance - #{amount},
    frozen_balance = frozen_balance + #{amount},
    version = version + 1
WHERE user_id = #{userId} AND version = #{version}
  AND available_balance >= #{amount}
```

### 7.1 多级缓存架构

```
┌─────────────────────────────────────────┐

│  │ 订单详情  │ 抢券库存  │ 发券锁   │     │

│  │ 验证码   │ 签到标记  │          │     │
│  └──────────┴──────────┴──────────┘     │
├─────────────────────────────────────────┤
│            本地缓存 (Caffeine)            │  ←── 类目树、字典配置
└─────────────────────────────────────────┘
```

### 7.2 缓存策略

| 数据类型 | 缓存策略 | TTL | Key格式 | 更新方式 |
|---------|---------|-----|---------|---------|
| 订单详情 | Cache Aside | 30min | `order:detail:{orderId}` | 写DB后删缓存 |
| 商品详情 | Cache Aside | 1h | `product:detail:{spuId}` | 写DB后删缓存 |
| 抢券库存 | 预加载 | 永久(扣完为止) | `coupon:stock:{templateId}` | Lua原子扣 |
| 用户Token | 黑名单 | 等于剩余有效期 | `token:blacklist:{tokenId}` | 登出时写入 |
| 验证码 | 自动过期 | 5min | `sms:code:{phone}` | 发送时写入 |
| 签到状态 | Bitmap | 永久 | `signin:{userId}:{yearMonth}` | setbit |
| 类目树 | 本地缓存 | 30min | — | 定时刷新 |
| 限流计数 | 滑动窗口 | 按窗口 | `rate:limit:{api}:{userId}` | Sentinel |

### 7.3 缓存一致性保障

- **读操作**：先读Redis → 命中返回 → 未命中查MySQL → 回写Redis（设随机TTL防雪崩）
- **写操作**：先更新MySQL → 删除Redis缓存（延迟双删可选项）
- **强一致场景**：用Lua脚本原子操作（如扣库存）
- **最终一致场景**：MQ异步 + 定时对账

### 7.4 缓存穿透/击穿/雪崩防护

| 问题 | 方案 |
|------|------|
| 穿透（查不存在的数据） | 布隆过滤器 + 空值缓存（TTL=1min） |
| 击穿（热点key过期） | Redisson分布式锁，只让一个线程查DB |
| 雪崩（大量key同时过期） | 过期时间 + 随机偏移（±30%） |

## 十、安全体系设计

### 10.1 认证流程

```
┌──────────┐                    ┌──────────┐
│  客户端   │                    │  服务端   │
└────┬─────┘                    └────┬─────┘
     │  1. 用户名+密码                 │
     │ ──────────────────────────►    │
     │                               │ 2. BCrypt校验
     │                               │ 3. 生成双Token
     │  4. Access Token(2h)          │
     │     + Refresh Token(7d)       │
     │ ◄───────────────────────────  │
     │                               │
     │  5. 请求(Header: Authorization │
     │     Bearer {accessToken})      │
     │ ──────────────────────────►    │
     │                               │ 6. JWT验签+解析
     │                               │ 7. 检查黑名单
     │                               │ 8. 注入SecurityContext
     │  9. Response                  │
     │ ◄───────────────────────────  │
     │                               │
     │ 10. AccessToken过期            │
     │ 11. 用RefreshToken刷新         │
     │ ──────────────────────────►    │
     │                               │ 12. 验RefreshToken
     │ 13. 新Access+Refresh          │
     │ ◄───────────────────────────  │
```

### 10.2 安全措施清单

| 层面 | 措施 | 实现方式 |
|------|------|---------|
| 传输层 | HTTPS强制 | Nginx SSL终结 |
| 认证 | JWT + RSA签名 | auth-service签发，gateway校验 |
| 密码 | BCrypt哈希 | Spring Security PasswordEncoder |
| 防暴力破解 | 登录失败计数 | Redis计数器 + 递增等待时间 |
| 防重放 | Token内嵌jti唯一ID + 时间戳 | JWT claim |
| 接口鉴权 | 注解驱动 `@PreAuthorize` | Spring Security方法级安全 |
| 数据隔离 | tenant_id过滤 | MyBatis-Plus拦截器自动注入 |
| SQL注入 | 参数化查询 | MyBatis-Plus默认防注入 |
| XSS | 输入过滤 | 全局过滤器HTML转义 |
| CSRF | SameSite Cookie + Token校验 | API层面Token免疫CSRF |
| 敏感数据 | 加密存储 | 手机号AES加密、日志脱敏 |
| API限流 | Sentinel | 接口级+用户级双层限流 |

### 10.3 租户隔离方案

```
┌──────────────────────────────────────────┐
│            多租户数据隔离策略                │
├──────────────────────────────────────────┤
│  共享数据库 + 共享表 + tenant_id字段隔离    │
│                                           │
│  所有业务表包含 tenant_id 字段              │
│  MyBatis-Plus 拦截器自动:                  │
│    1. SELECT → WHERE tenant_id = ?       │
│    2. INSERT → set tenant_id             │
│    3. UPDATE → WHERE tenant_id = ?       │
│    4. DELETE → WHERE tenant_id = ?       │
│                                           │
│  租户ID来源:                               │
│    Header: X-Tenant-Id                   │
│    (由Gateway解析Token后注入)              │
└──────────────────────────────────────────┘
```
