## 一、技术选型差异

| 架构书 | 实际 | 原因 | 架构书需改？ |
|--------|------|------|-------------|
| ShardingSphere-**JDBC** 5.4.x | ShardingSphere-**Proxy** 5.5.3 | JDBC内嵌模式SnakeYAML冲突无解，Proxy旁路更干净 | ✅ 是 |
| **Redisson** 3.27.x | StringRedisTemplate.setIfAbsent | Windows Docker下Netty连不上Redis | ✅ 是 |
| **Sentinel** 1.8.6 | 无 | 开发阶段未接入，Gateway层无限流熔断 | ✅ 是 |
| **Seata** 1.8.x | 无 | 用MQ+定时对账做最终一致，未接分布式事务 | ✅ 是 |
| **XXL-JOB** 2.4.0 | Spring @Scheduled | 开发阶段用内置定时器替代，无调度面板 | ✅ 是 |
| **MapStruct** 1.5.x | 无 | 代码中手动setter或Hutool转换 | 🟢 可保留 |
| **DFA敏感词过滤** | 无 | IM消息无内容审核 | ✅ 是 |
| **Canal server mode** | `kafka`(文档写) → `rabbitMQ`(实际) | 架构书示例用的是Kafka，实际用的RabbitMQ | ✅ 是 |

## 四、MQ消费者/生产者差异

| 架构书描述 | 实际 |
|-----------|------|
| `order.create.queue` → notify-service, data-service | ❌ 无消费者监听（我们补了OrderEventConsumer，但监听的PAYMENT_CALLBACK_QUEUE） |
| `order.cancel.queue` → coupon-service, payment-service | ❌ 无消费者 |
| `payment.callback.queue` → order-service, settlement-service | ⚠️ OrderEventConsumer监听，但settlement未监听此队列 |
| `notify.*.queue` → notify-service | ⚠️ NotifyService监听NOTIFY_INBOX_QUEUE，但无人发送 |
| `settlement.settle.queue` → settlement-service | ✅ SettlementEventConsumer已补 |
| `im.*.queue` → im-service | ❌ 无消费者 |

## 五、总结

| 类别 | 数量 |
|------|------|
| 技术选型差异 | 8 项 |
| 各服务功能差异 | 约 55 项（已实现 vs 架构书声称） |
| 缺失数据库表 | 约 25 张 |
| MQ链路未完成 | 4 条（共约10条） |

> **结论**：代码骨架完整，15个服务+Gateway+中间件全部跑通，核心业务链路（下单→支付→分账→搜索同步）验证通过。架构书需要更新以反映实际的技术选型和已完成的功能范围，未实现的功能标注为后续迭代计划。
