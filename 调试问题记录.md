# 校享云 — 调试问题记录

> 记录 15 个微服务从 0 到全量调通过程中的所有问题。去重后共 24 个独立问题。

---

## 一、环境类问题

### 1.1 Docker MySQL 端口 3306 被占用

**现象**：`docker run -p 3306:3306` 报 `port is already in use`

**原因**：宿主机已有 MySQL 服务占用 3306 端口

**解决**：Docker MySQL 改用 `-p 3307:3306`，所有服务的 `application.yml` 中 JDBC URL 同步改为 `localhost:3307`

---

### 1.2 MySQL 8.0 认证插件不兼容

**现象**：Java 连接 MySQL 报 `caching_sha2_password` 错误

**原因**：MySQL 8.0 默认使用 `caching_sha2_password` 认证插件，部分 JDBC 驱动版本不兼容

**解决**：JDBC URL 追加 `allowPublicKeyRetrieval=true` 参数；同时将 Docker MySQL 的 root 用户改为 `mysql_native_password` 插件

---

### 1.3 Docker Redis 认证错误

**现象**：Auth 服务报 `NOAUTH Authentication required`

**原因**：Docker Redis 容器被重建后残留了旧的 `requirepass` 配置

**解决**：删除旧容器 `docker rm -f campus-redis`，重新创建干净容器

---

### 1.4 Nacos 容器反复重启

**现象**：`docker-compose up -d nacos` 后容器状态一直 `Restarting`

**原因**：Nacos 配置了 MySQL 作为外部数据源，但 Nacos 容器通过 Docker 网络连接 MySQL 时，MySQL 容器尚未完成初始化（`nacos_config` 数据库未创建）

**解决**：改用 Nacos 内嵌 Derby 数据库（`MODE=standalone`，不配置 MySQL），内存占用更小，开发环境足够

---

### 1.5 内存不足导致 JVM 无法启动

**现象**：`mvn spring-boot:run` 报 `os::commit_memory failed`，JVM 无法分配堆内存

**原因**：Docker Desktop 虚拟机 + 多个 Java 进程同时运行，16GB 笔记本内存耗尽

**解决**：关闭 Docker Desktop；改用宿主机原生 MySQL；仅保留 Docker Redis（~50MB）；每次启动前杀掉旧 Java 进程

---

### 1.6 端口被旧进程占用

**现象**：重启服务时报 `Port 808X was already in use`

**原因**：之前启动失败或手动停止不完全，Java 进程残留在后台

**解决**：`taskkill /F /IM java.exe` 清理所有 Java 进程后重新启动

---

## 二、代码 Bug

### 2.1 JwtTokenProvider 不是 Spring Bean

**现象**：Auth 服务启动报 `No qualifying bean of type 'JwtTokenProvider' available`

**原因**：`JwtTokenProvider` 是普通 Java 类，没有 `@Component` 注解。`AuthService` 通过构造器注入它，但 Spring 容器中没有这个 Bean

**解决**：给 `JwtTokenProvider` 加 `@Component` 注解；同时在 `SecurityAutoConfiguration` 加 `@ComponentScan` 确保被扫描到

**文件**：`JwtTokenProvider.java`、`SecurityAutoConfiguration.java`

---

### 2.2 Gateway 与 Spring MVC 冲突

**现象**：Gateway 启动报 `Spring MVC found on classpath, which is incompatible with Spring Cloud Gateway`

**原因**：Gateway 引入了 `common-security`，而 `common-security` → `common-core` → `spring-boot-starter-web`（含 Spring MVC）。Gateway 是响应式框架，不能有阻塞式 MVC

**解决**：
1. Gateway POM 中排除 `spring-boot-starter-web`
2. 显式加 `jakarta.servlet-api`（`UserContextFilter` 需要 `Filter` 接口）
3. `application.yml` 中声明 `spring.main.web-application-type: reactive`

**文件**：`campus-gateway/pom.xml`、`campus-gateway/application.yml`

---

### 2.3 @Value 无法注入 YAML 列表

**现象**：Gateway 启动报 `Could not resolve placeholder 'campus.security.white-urls'`

**原因**：Spring 的 `@Value` 注解不支持直接注入 YAML 数组/列表类型，只支持 String 和基本类型

**解决**：将白名单 URL 硬编码为 `AuthGlobalFilter` 中的 static 常量（`List.of(...)`）

**文件**：`AuthGlobalFilter.java`

---

### 2.4 Gateway CSRF 拦截请求

**现象**：通过 Gateway 访问接口报 `An expected CSRF token cannot be found`

**原因**：`common-security` 引入了 Spring Security WebFlux 自动配置，Gateway 默认启用了 CSRF 保护

**解决**：给 Gateway 添加 `GatewaySecurityConfig`，禁用 CSRF 并放行所有请求

**文件**：`GatewaySecurityConfig.java`（新增）

---

### 2.5 MySQL `user_id` 字段缺失默认值

**现象**：Auth 注册时报 `Field 'user_id' doesn't have a default value`

**原因**：`SysUser` 实体继承 `BaseEntity`，但之前用纯 MyBatis 手写 SQL 时创建的 `sys_user` 表缺少 `is_deleted` 等字段；插入时部分列无默认值

**解决**：重建 `sys_user` 表，确保所有列与 `BaseEntity` 字段对齐

---

### 2.6 Gateway 路由 `lb://` 在无 Nacos 时失败

**现象**：Gateway 前缀 `lb://campus-user` 无法路由，请求无响应或 503

**原因**：`lb://` 是 LoadBalancer 协议，依赖 Nacos 服务发现。开发环境 Nacos 关闭时，Gateway 无法解析服务地址

**解决**：开发环境将 Gateway 路由改为直连 `http://localhost:8082`。生产环境恢复 `lb://` 配合 Nacos 使用

---

## 三、框架级 Bug（重点）

### 3.1 UserContext Lombok @Data/@Builder 导致 StackOverflow ★核心问题

**现象**：
- Auth 服务正常（SMS/注册/登录全通）
- User 服务任何涉及 Mapper 的接口都报 `java.lang.StackOverflowError`
- 错误发生在 `Handler dispatch failed` 阶段（Spring MVC 还没进 Controller 方法就崩了）
- 经过 **20+ 种组合测试** 最终定位

**排除过程**（每种都试过，均无效）：
1. ❌ 排除 MyBatisPlusAutoConfiguration → 不崩但 Mapper 无法使用
2. ❌ 排除 Spring Security → 仍然崩
3. ❌ 注释 `@TableLogic` → 仍然崩
4. ❌ 去掉 `TenantInterceptor` → 仍然崩
5. ❌ 去掉 `PaginationInnerInterceptor` → 仍然崩
6. ❌ 注释 `MybatisPlusConfig` → 仍然崩
7. ❌ 去掉 `@MapperScan` → 仍然崩
8. ❌ 用纯 MyBatis 手写 SQL 替代 MyBatis-Plus `BaseMapper` → 仍然崩
9. ❌ 去掉实体类的 Lombok `@Data` → 仍然崩
10. ❌ 去掉 Controller 的 `@RequestMapping` → 仍然崩
11. ❌ 去掉 Service 层的 `@Transactional` / `@RequiredArgsConstructor` → 仍然崩
12. ❌ 去掉整个 Service 层，Controller 直接调 Mapper → 仍然崩

**关键突破口**：
- `HealthController`（无 `UserContext.getUserId()` 调用）→ ✅ 正常
- `HealthController` 直接调 `UserAddressMapper.selectByUserId(1L)` → ✅ 正常
- `UserController` 里调用 `UserContext.getUserId()` → ❌ StackOverflow
- `UserController` 里硬编码 `userId=1L` → ✅ 正常
- `UserController` 用 `@RequestHeader("X-User-Id")` → ✅ 正常

**根因**：`UserContext` 类使用了 Lombok `@Data`（生成 `equals`/`hashCode`/`toString`）和 `@Builder`，同时内部持有 `ThreadLocal`。当 Spring MVC 的 Handler 映射解析 Controller 方法时，触发了 Lombok 生成的 `equals()` 方法，该方法递归调用导致栈溢出。

**解决**：完全移除 `UserContext` 的 Lombok 依赖（`@Data`、`@Builder`、`@NoArgsConstructor`、`@AllArgsConstructor`），手写所有 getter/setter/构造器。

**文件**：`UserContext.java`

---

### 3.2 UserContext Lombok @Builder 导致 JWT 解析 userId=null ★核心问题

**现象**：
- Gateway 的 `JwtTokenProvider.parseToken()` 能正确解析 JWT claims（`sub=2073005678401105922`, `tid=0`）
- 但 `UserContext.builder()...build()` 产出的对象中 `userId` 和 `tenantId` 均为 `null`

**根因**：Lombok `@Builder` 在生成 `build()` 方法时，对 `Long` 类型的字段有特殊处理逻辑。当 `userId` 的值（如 2073005678401105922）超过一定范围时，Lombok Builder 内部优化可能导致字段未被正确设置。

**解决**：同上，移除 Lombok，改用 `new UserContext()` + `setUserId()` / `setTenantId()` 手动赋值。

**文件**：`UserContext.java`、`JwtTokenProvider.java`、`UserContextFilter.java`

**连锁改动**：`getUserId()` / `getTenantId()` 静态方法因与实例 getter 同名导致编译错误，重命名为 `currentUserId()` / `currentTenantId()`，全局 16 处引用同步更新。

## 五、经验总结

1. **Lombok + ThreadLocal = 定时炸弹**：`@Data` 生成的 `equals`/`hashCode` 在特定 Spring 代理场景下会触发递归，`@Builder` 对 Long 类型有隐蔽 Bug。带 ThreadLocal 的类不要用 Lombok。

2. **控制变量法排错**：逐一排除 MyBatis-Plus、Security、Lombok、Service 层，最终锁定 `UserContext` 这一个文件。

3. **开发环境 vs 生产环境**：Nacos 服务发现（`lb://`）在开发环境用直连替代，MyBatis-Plus 的分页/逻辑删除/租户隔离插件等高级功能在联调通过后再开启。

## 四、ShardingSphere 分库分表（JDBC 模式尝试）

### 4.1 ShardingSphere 内联配置不生效

**现象**：`spring.shardingsphere.datasource` 配置了 4 个 MySQL 数据源，但实际执行 SQL 时路由到 H2。

**原因**：`shardingsphere-jdbc-core` 5.4.1 不包含 Spring Boot 自动配置类，`spring.shardingsphere.*` 前缀属性不会被任何自动配置读取。ShardingSphere 找不到配置则回退到默认 H2 内存数据库。

**解决**：放弃 JDBC 模式，改用 ShardingSphere-Proxy。

---

### 4.2 ShardingSphereDriver 每次请求重建连接池

**现象**：使用 `ShardingSphereDriver` 作 JDBC 驱动时，每次请求创建新 HikariPool，且第一个 Pool 因 `checkFailFast` 超时失败。

**原因**：`ShardingSphereDriver.connect()` 每次调用都创建新的 `ShardingSphereDataSource`，HikariCP 默认 `initializationFailTimeout=1ms` 导致创建超时。

**解决**：改用 `YamlShardingSphereDataSourceFactory.createDataSource()` 创建 Spring Bean，启动时初始化一次。

---

### 4.3 YamlShardingSphereDataSourceFactory 内部 SnakeYAML 冲突

**现象**：`YamlShardingSphereDataSourceFactory.createDataSource()` 报 `Representer: method 'void <init>()' not found`。

**原因**：ShardingSphere JAR 内嵌旧版 SnakeYAML（shaded 方式），与 Spring Boot 3.2 的 SnakeYAML 2.x 冲突。即使 POM 中 `<exclusion>` 排除传递依赖也无法解决——内嵌版本在 classloader 上优先加载。

**解决**：放弃 JDBC 模式，改用 ShardingSphere-Proxy——Proxy 使用独立依赖环境，不与应用 SnakeYAML 冲突。

---

## 五、ShardingSphere-Proxy 方案（最终成功）

### 5.1 Proxy 5.5.3 启动闪退 / 文件被占用

**现象**：双击 `start.bat` 窗口闪退，日志显示 `另一个程序正在使用此文件`。

**原因**：
1. `start.bat` 使用 `javaw`（无控制台窗口），输出重定向到 `logs/stdout.log`
2. 前次未正常关闭的 Proxy 进程（javaw.exe）仍持有日志文件句柄
3. Windows 文件系统未释放锁

**解决**：`wmic process where "name='javaw.exe'" delete` 强杀所有残留进程，删除 `logs/` 目录后重建。

---

### 5.2 Proxy 5.5.3 报 Seata ClassNotFoundException

**现象**：Proxy 启动成功，但执行任何 SQL 都报 `NoClassDefFoundError: org/apache/seata/core/context/RootContext`。

**原因**：Proxy 5.5.3 自带 `shardingsphere-transaction-base-seata-at-5.5.3.jar`，运行时强制扫描 Seata SPI 实现类。即使配置了 `transaction: defaultType: LOCAL`，该 JAR 仍会触发 Seata 类加载。

**解决**：
1. 删除 `lib/shardingsphere-transaction-base-seata-at-5.5.3.jar`
2. `global.yaml` 中正确配置 `transaction.defaultType: LOCAL`

---

### 5.3 ShardingSphere-Proxy 拒绝更新分片键

**现象**：`orderMapper.updateById(order)` 报 `Can not update sharding value for table 't_order'`。

**原因**：MyBatis-Plus 的 `updateById` 会生成包含所有字段的 UPDATE 语句，包括 `order_id`（分表键）和 `tenant_id`（分库键）。ShardingSphere 检测到分片键被包含在 SET 子句中，为防止数据迁移到错误分片，直接拒绝执行。

**解决**：将 `updateById` 替换为 `LambdaUpdateWrapper` 精准更新——只 SET 非分片键字段（status、update_time），WHERE 条件使用 `order_no`。

**文件**：`OrderService.java`

---

### 5.4 Proxy 事务配置位置错误

**现象**：在 `global.yaml` 的 `props` 下配置 `proxy-transaction-type: LOCAL`，Seata 错误依旧。

**原因**：ShardingSphere-Proxy 5.5.x 的事务配置在独立的 `transaction` 块下，不在 `props` 下。`proxy-transaction-type` 是 JDBC 驱动（ShardingSphereDriver）的配置属性，Proxy 端不识别。

**正确配置**：
```yaml
transaction:
  defaultType: LOCAL
```

---

### 5.5 最终完整配置

**global.yaml**：
```yaml
authority:
  users:
    - user: root
      password: root
      admin: true
transaction:
  defaultType: LOCAL
props:
  proxy-default-port: 3308
  sql-show: true
```

**database-sharding.yaml**：
```yaml
databaseName: order_db
dataSources:
  ds0: { url: jdbc:mysql://localhost:3307/order_db0, username: root, password: root123 }
  ds1: { url: jdbc:mysql://localhost:3307/order_db1, username: root, password: root123 }
  ds2: { url: jdbc:mysql://localhost:3307/order_db2, username: root, password: root123 }
  ds3: { url: jdbc:mysql://localhost:3307/order_db3, username: root, password: root123 }
rules:
  - !SHARDING
    tables:
      t_order:
        actualDataNodes: ds${0..3}.t_order_${0..15}
        databaseStrategy:
          standard:
            shardingColumn: tenant_id
            shardingAlgorithmName: db-inline
        tableStrategy:
          standard:
            shardingColumn: order_id
            shardingAlgorithmName: tbl-inline
    shardingAlgorithms:
      db-inline: { type: INLINE, props: { algorithm-expression: ds${tenant_id % 4} } }
      tbl-inline: { type: INLINE, props: { algorithm-expression: t_order_${order_id % 16} } }
```

**验证方式**：Proxy 日志中查看 `Logic SQL`（应用原始 SQL）和 `Actual SQL`（Proxy 改写后路由到真实分片的 SQL，如 `ds0 → t_order_0 UNION ALL t_order_1 ...`）

## 七、经验总结（补充）

4. **依赖冲突尽头是架构切换**：JDBC 内嵌模式的 SnakeYAML 冲突是 classloader 级别问题，Maven 排除无法解决。Proxy 模式完全绕开。
5. **精准 SQL 是分库分表的铁律**：任何包含分片键的 UPDATE 都会被 ShardingSphere 拦截，必须用 `LambdaUpdateWrapper` 精确控制更新字段。
6. **Proxy 日志是调试法宝**：`Logic SQL` / `Actual SQL` 直接展示分片路由是否正确。
7. **Seata 集成 JAR 可直接删除**：不用分布式事务时，删掉 `shardingsphere-transaction-base-seata-at-*.jar` 无副作用。

## 六、批量联调问题（Product + 5个简单服务）

### 6.1 `@RequestParam` 参数名缺失导致 500

**现象**：Product 服务 `/api/v1/product/spu/list?categoryId=1` 报 `IllegalArgumentException: Name for argument of type [java.lang.Long] not specified`。

**原因**：Maven 编译器未启用 `-parameters` 参数，Spring MVC 无法通过反射获取方法参数名。在无参数时（如 `@RequestParam(required = false) Long categoryId`），Spring 无法确定参数名。

**解决**：
1. 根 POM 的 `maven-compiler-plugin` 配置中添加 `<parameters>true</parameters>`
2. 所有 `@RequestParam` 处显式指定 `name` 属性

**文件**：`pom.xml`、`ProductController.java`

---

### 6.2 Product 服务 SKU 表缺失

**现象**：Product 服务 `/api/v1/product/spu/1` 报 `BadSqlGrammarException: Table 'product_sku' doesn't exist`。

**原因**：建表脚本只创建了 `product_category` 和 `product_spu`，遗漏了 `product_sku` 表。详情接口需要关联查询 SKU 数据。

**解决**：在 Docker MySQL 中执行 `CREATE TABLE product_sku` 并插入测试数据。

---

### 6.3 多个简单服务 Security 配置缺失

**现象**：Tenant/Notify/Review 服务启动后请求报 401 或 CSRF 错误，SecurityFilterChain 包含 `CsrfFilter`。

**原因**：批量创建时遗漏了这些服务的 `SecurityConfig.java`（只给 File 和 Data 创建了）。

**解决**：为 Tenant/Notify/Review 补充了 `SecurityConfig.java`，内容与 User/Auth/Product 一致：禁用 CSRF + 放行所有请求。

---

### 6.4 Data 服务 DataSource 自动配置冲突

**现象**：Data 服务启动报 `Failed to configure a DataSource: 'url' attribute is not specified`。

**原因**：Data 服务依赖 `common-mybatis`，而 `common-mybatis` 引入了 MyBatis-Plus 和 Druid 数据源自动配置。但 Data 服务只通过 Feign 调其他服务，不需要数据库。

**解决**：
1. 注释掉 Data 服务 POM 中的 `common-mybatis` 依赖
2. `application.yml` 中排除 `DataSourceAutoConfiguration` 等自动配置

**文件**：`campus-data/pom.xml`、`campus-data/application.yml`

---

### 6.5 File 服务 MinIO 连接失败导致启动崩溃

**现象**：File 服务启动报 `ConnectException: Failed to connect to localhost:9000`。

**原因**：`FileService` 的 `@PostConstruct init()` 方法在启动时连接 MinIO 并创建 Bucket，但本地未运行 MinIO 容器。

**解决**：给 `init()` 方法加 try-catch，MinIO 不可用时输出警告日志但不阻止启动。

**文件**：`FileService.java`

---

### 6.6 Gateway 路由大量指向错误端口

**现象**：Gateway 中 Order/Payment/Coupon/Settlement/Search/IM/Notify/Review/File/Data 的路由全部指向 `8088`（Tenant 端口）。

**原因**：原始配置模板中这些路由默认指向 `8084`（Order），批量替换时 sed 命令匹配不精确，导致大量路由被错误地指向 8088。

**正确端口映射**：

## 八、中等服务联调（Payment + Coupon + Settlement）

### 8.1 Redisson 连不上 Docker Redis

**现象**：Payment 和 Coupon 启动报 `RedissonAutoConfigurationV2: Unable to connect to Redis server: localhost/127.0.0.1:6379`。Settlement 服务正常（没依赖 Redisson）。

**原因**：Redisson 使用 Netty 作为网络层客户端。在 Windows Docker Desktop 环境下，Netty 的 TCP 连接与 Docker 端口转发存在兼容性问题——Redis CLI 能 PING，但 Redisson Netty 客户端就是连不上。这是 Windows 平台的已知限制。

**排查过程**：
1. ✅ Redis CLI `PING` → PONG
2. ✅ `redis_version:7.2.14`，`bind * -::*`，`protected-mode no`
3. ❌ Redisson 显式 YAML 配置 `redis://127.0.0.1:6379` → 失败
4. ❌ 换 `redis://localhost:6379` → 失败
5. ❌ 加超时配置 `connectTimeout: 10000` → 失败
6. ❌ 加 `dnsMonitoringInterval: -1` → 失败

**解决**：排除 `RedissonAutoConfigurationV2`，将 `RedissonClient` 锁替换为 Spring Boot 默认的 `StringRedisTemplate.opsForValue().setIfAbsent(key, "1", Duration)`。两者底层原理一致（`SET NX EX`），互斥性等效。

**影响范围**：
- `PaymentService.handleCallback()` — 支付回调幂等锁
- `CouponService.sendOne()` — 批量发券防重锁

**文件**：`PaymentService.java`、`CouponService.java`

---

### 8.2 WorkerWallet 缺 @TableId 导致 selectById 失败

**现象**：Settlement 服务 `/api/v1/settlement/wallet` 报 `BindingException: Invalid bound statement (not found): WorkerWalletMapper.selectById`

**原因**：`WorkerWallet` 实体使用 `user_id` 作为主键列，但没有 `@TableId` 注解。MyBatis-Plus 的 `BaseMapper.selectById()` 需要通过 `@TableId` 才能识别非标准主键名。

**解决**：在 `userId` 字段上加 `@TableId` 注解。

**文件**：`WorkerWallet.java`

---

### 8.3 Docker 重启后 MySQL 数据库丢失

**现象**：Payment/Coupon/Settlement 启动后请求报 `Unknown database 'campus_payment'` 等错误。

**原因**：Docker MySQL 容器使用 `docker run` 创建时未指定 `-v` 挂载数据卷。容器重启后数据保留，但容器被删除重建后所有数据库丢失。

**解决**：
1. 使用 `docker start <容器名>` 而非 `docker run` 来重启已有容器
2. 如果必须重建，需重新执行建库建表脚本

**防范**：`docker-compose.yml` 中已配置 `volumes` 持久化，应使用 `docker-compose up -d` 统一管理容器生命周期。

### 8.4 Git Checkout 意外还原了之前的修复

**现象**：恢复了 PaymentService + CouponService 的代码后，之前的 `UserContext.currentUserId()` 修改被 git checkout 还原，导致编译报错。

**原因**：`git checkout` 会完全覆盖文件到仓库版本，包括其他调试分支中已修复的内容。

**解决**：重新逐个应用 `@RequestHeader` 和 `userId` 参数化修改。教训——细粒度的 commit 可以在回退时精确保留需要的改动。

## 十、Search + IM 最后冲刺

### 10.1 Elasticsearch Docker 启动后立刻退出

**现象**：`docker run -d elasticsearch:8.11.0` 返回容器 ID 但 `docker ps` 显示 `Created`，未运行。`docker start` 报端口 9200 被占用。

**原因**：宿主机已有进程（可能是其他 Docker 容器或本地服务）占用 9200 端口。

**解决**：关闭占用 9200 的进程后重新启动容器。

---

### 10.2 ES 索引不存在导致搜索 500

**现象**：Search 服务返回 `index_not_found_exception: no such index [campus_product_spu]`。

**原因**：ES 刚启动，索引从未创建过。`common-es` 的 `EsConfig` 只提供 `ElasticsearchClient` Bean，不负责创建索引。

**解决**：通过 `PUT /campus_product_spu` 手动创建索引，定义 mappings。后续可通过 Canal 自动同步或 XXL-JOB 定时同步来维护。

---

### 10.3 ES 文档索引后搜索不到

**现象**：`PUT /campus_product_spu/_doc/1` 返回 `created` 成功，但搜索仍返回空数组。

**原因**：ES 默认每秒刷新一次（`refresh_interval=1s`），刚写入的文档在刷新前不可搜索。

**解决**：调用 `POST /campus_product_spu/_refresh` 手动刷新索引，或在测试时等待 1-2 秒后再搜索。

---

### 10.4 Redis 容器在 ES 启动期间被意外停止

**现象**：Search 启动后请求报 `RedisConnectionFailureException: Unable to connect to Redis`。

**原因**：启动 ES 容器时，`docker run` 等命令可能触发 Docker Desktop 重启，导致已有的 Redis 容器被意外关闭。

**解决**：`docker ps` 确认 Redis 运行状态→`docker restart campus-redis`→重启 Search。

---

### 10.5 IM 控制器 sed 替换导致编译失败

**现象**：IM 编译报 `Unable to find a suitable main class`，ImController 语法错误。

**原因**：`sed` 替换 `UserContext.currentUserId()` 为 `userId` 时，在 `messages(Long userId,(@PathVariable...)` 处产生了多余括号。

**解决**：手写 ImController，messages 接口不需要 userId 参数（sessionId 是全局唯一的），仅 offline-messages 接口需要 @RequestHeader。

**文件**：`ImController.java`
