# 校享云 — 调试问题记录

> 记录从首次启动到 Gateway + Auth + User 完整链路调通过程中遇到的所有问题。

---

## 一、环境类问题

### 1.1 Docker MySQL 端口 3306 被占用

**现象**：`docker run -p 3306:3306` 报 `port is already in use`

**原因**：宿主机已有 MySQL 服务占用 3306 端口

**解决**：Docker MySQL 改用 `-p 3307:3306`，所有服务的 `application.yml` 中 JDBC URL 同步改为 `localhost:3307`

---

### 1.2 MySQL 8.0 认证插件不兼容

**现象**：Java 连接 MySQL 报 `caching_sha2_password` 错误

**原因**：MySQL 8.0 默认使用 `caching_sha2_password` 认证插件，部分 JDBC 驱动版本不兼容

**解决**：JDBC URL 追加 `allowPublicKeyRetrieval=true` 参数；同时将 Docker MySQL 的 root 用户改为 `mysql_native_password` 插件

---

### 1.3 Docker Redis 认证错误

**现象**：Auth 服务报 `NOAUTH Authentication required`

**原因**：Docker Redis 容器被重建后残留了旧的 `requirepass` 配置

**解决**：删除旧容器 `docker rm -f campus-redis`，重新创建干净容器

---

### 1.4 Nacos 容器反复重启

**现象**：`docker-compose up -d nacos` 后容器状态一直 `Restarting`

**原因**：Nacos 配置了 MySQL 作为外部数据源，但 Nacos 容器通过 Docker 网络连接 MySQL 时，MySQL 容器尚未完成初始化（`nacos_config` 数据库未创建）

**解决**：改用 Nacos 内嵌 Derby 数据库（`MODE=standalone`，不配置 MySQL），内存占用更小，开发环境足够

---

### 1.5 内存不足导致 JVM 无法启动

**现象**：`mvn spring-boot:run` 报 `os::commit_memory failed`，JVM 无法分配堆内存

**原因**：Docker Desktop 虚拟机 + 多个 Java 进程同时运行，16GB 笔记本内存耗尽

**解决**：关闭 Docker Desktop；改用宿主机原生 MySQL；仅保留 Docker Redis（~50MB）；每次启动前杀掉旧 Java 进程

---

### 1.6 端口被旧进程占用

**现象**：重启服务时报 `Port 808X was already in use`

**原因**：之前启动失败或手动停止不完全，Java 进程残留在后台

**解决**：`taskkill /F /IM java.exe` 清理所有 Java 进程后重新启动

---

## 二、代码 Bug

### 2.1 JwtTokenProvider 不是 Spring Bean

**现象**：Auth 服务启动报 `No qualifying bean of type 'JwtTokenProvider' available`

**原因**：`JwtTokenProvider` 是普通 Java 类，没有 `@Component` 注解。`AuthService` 通过构造器注入它，但 Spring 容器中没有这个 Bean

**解决**：给 `JwtTokenProvider` 加 `@Component` 注解；同时在 `SecurityAutoConfiguration` 加 `@ComponentScan` 确保被扫描到

**文件**：`JwtTokenProvider.java`、`SecurityAutoConfiguration.java`

---

### 2.2 Gateway 与 Spring MVC 冲突

**现象**：Gateway 启动报 `Spring MVC found on classpath, which is incompatible with Spring Cloud Gateway`

**原因**：Gateway 引入了 `common-security`，而 `common-security` → `common-core` → `spring-boot-starter-web`（含 Spring MVC）。Gateway 是响应式框架，不能有阻塞式 MVC

**解决**：
1. Gateway POM 中排除 `spring-boot-starter-web`
2. 显式加 `jakarta.servlet-api`（`UserContextFilter` 需要 `Filter` 接口）
3. `application.yml` 中声明 `spring.main.web-application-type: reactive`

**文件**：`campus-gateway/pom.xml`、`campus-gateway/application.yml`

---

### 2.3 @Value 无法注入 YAML 列表

**现象**：Gateway 启动报 `Could not resolve placeholder 'campus.security.white-urls'`

**原因**：Spring 的 `@Value` 注解不支持直接注入 YAML 数组/列表类型，只支持 String 和基本类型

**解决**：将白名单 URL 硬编码为 `AuthGlobalFilter` 中的 static 常量（`List.of(...)`）

**文件**：`AuthGlobalFilter.java`

---

### 2.4 Gateway CSRF 拦截请求

**现象**：通过 Gateway 访问接口报 `An expected CSRF token cannot be found`

**原因**：`common-security` 引入了 Spring Security WebFlux 自动配置，Gateway 默认启用了 CSRF 保护

**解决**：给 Gateway 添加 `GatewaySecurityConfig`，禁用 CSRF 并放行所有请求

**文件**：`GatewaySecurityConfig.java`（新增）

---

### 2.5 MySQL `user_id` 字段缺失默认值

**现象**：Auth 注册时报 `Field 'user_id' doesn't have a default value`

**原因**：`SysUser` 实体继承 `BaseEntity`，但之前用纯 MyBatis 手写 SQL 时创建的 `sys_user` 表缺少 `is_deleted` 等字段；插入时部分列无默认值

**解决**：重建 `sys_user` 表，确保所有列与 `BaseEntity` 字段对齐

---

### 2.6 Gateway 路由 `lb://` 在无 Nacos 时失败

**现象**：Gateway 前缀 `lb://campus-user` 无法路由，请求无响应或 503

**原因**：`lb://` 是 LoadBalancer 协议，依赖 Nacos 服务发现。开发环境 Nacos 关闭时，Gateway 无法解析服务地址

**解决**：开发环境将 Gateway 路由改为直连 `http://localhost:8082`。生产环境恢复 `lb://` 配合 Nacos 使用

---

## 三、框架级 Bug（重点）

### 3.1 UserContext Lombok @Data/@Builder 导致 StackOverflow ★核心问题

**现象**：
- Auth 服务正常（SMS/注册/登录全通）
- User 服务任何涉及 Mapper 的接口都报 `java.lang.StackOverflowError`
- 错误发生在 `Handler dispatch failed` 阶段（Spring MVC 还没进 Controller 方法就崩了）
- 经过 **20+ 种组合测试** 最终定位

**排除过程**（每种都试过，均无效）：
1. ❌ 排除 MyBatisPlusAutoConfiguration → 不崩但 Mapper 无法使用
2. ❌ 排除 Spring Security → 仍然崩
3. ❌ 注释 `@TableLogic` → 仍然崩
4. ❌ 去掉 `TenantInterceptor` → 仍然崩
5. ❌ 去掉 `PaginationInnerInterceptor` → 仍然崩
6. ❌ 注释 `MybatisPlusConfig` → 仍然崩
7. ❌ 去掉 `@MapperScan` → 仍然崩
8. ❌ 用纯 MyBatis 手写 SQL 替代 MyBatis-Plus `BaseMapper` → 仍然崩
9. ❌ 去掉实体类的 Lombok `@Data` → 仍然崩
10. ❌ 去掉 Controller 的 `@RequestMapping` → 仍然崩
11. ❌ 去掉 Service 层的 `@Transactional` / `@RequiredArgsConstructor` → 仍然崩
12. ❌ 去掉整个 Service 层，Controller 直接调 Mapper → 仍然崩

**关键突破口**：
- `HealthController`（无 `UserContext.getUserId()` 调用）→ ✅ 正常
- `HealthController` 直接调 `UserAddressMapper.selectByUserId(1L)` → ✅ 正常
- `UserController` 里调用 `UserContext.getUserId()` → ❌ StackOverflow
- `UserController` 里硬编码 `userId=1L` → ✅ 正常
- `UserController` 用 `@RequestHeader("X-User-Id")` → ✅ 正常

**根因**：`UserContext` 类使用了 Lombok `@Data`（生成 `equals`/`hashCode`/`toString`）和 `@Builder`，同时内部持有 `ThreadLocal`。当 Spring MVC 的 Handler 映射解析 Controller 方法时，触发了 Lombok 生成的 `equals()` 方法，该方法递归调用导致栈溢出。

**解决**：完全移除 `UserContext` 的 Lombok 依赖（`@Data`、`@Builder`、`@NoArgsConstructor`、`@AllArgsConstructor`），手写所有 getter/setter/构造器。

**文件**：`UserContext.java`

---

### 3.2 UserContext Lombok @Builder 导致 JWT 解析 userId=null ★核心问题

**现象**：
- Gateway 的 `JwtTokenProvider.parseToken()` 能正确解析 JWT claims（`sub=2073005678401105922`, `tid=0`）
- 但 `UserContext.builder()...build()` 产出的对象中 `userId` 和 `tenantId` 均为 `null`

**根因**：Lombok `@Builder` 在生成 `build()` 方法时，对 `Long` 类型的字段有特殊处理逻辑。当 `userId` 的值（如 2073005678401105922）超过一定范围时，Lombok Builder 内部优化可能导致字段未被正确设置。

**解决**：同上，移除 Lombok，改用 `new UserContext()` + `setUserId()` / `setTenantId()` 手动赋值。

**文件**：`UserContext.java`、`JwtTokenProvider.java`、`UserContextFilter.java`

**连锁改动**：`getUserId()` / `getTenantId()` 静态方法因与实例 getter 同名导致编译错误，重命名为 `currentUserId()` / `currentTenantId()`，全局 16 处引用同步更新。

## 五、经验总结

1. **Lombok + ThreadLocal = 定时炸弹**：`@Data` 生成的 `equals`/`hashCode` 在特定 Spring 代理场景下会触发递归，`@Builder` 对 Long 类型有隐蔽 Bug。带 ThreadLocal 的类不要用 Lombok。

2. **控制变量法排错**：逐一排除 MyBatis-Plus、Security、Lombok、Service 层，最终锁定 `UserContext` 这一个文件。

3. **开发环境 vs 生产环境**：Nacos 服务发现（`lb://`）在开发环境用直连替代，MyBatis-Plus 的分页/逻辑删除/租户隔离插件等高级功能在联调通过后再开启。
