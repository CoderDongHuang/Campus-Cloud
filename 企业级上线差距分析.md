# 校享云 — 企业级上线差距分析

> 对比对象：当前项目（含 26 项迭代优化完成后） vs 真正可以上线的企业级 SaaS 平台
> 日期：2026-07-07

---

## 阅读指引

本文档覆盖 6 个维度共 20 项差距，每项包含：
- **差距描述**：当前状态 vs 生产要求
- **后果**：不上线会怎样
- **怎么改**：具体操作步骤、推荐技术方案
- **优先级**：🔴 上线前必须做 | 🟡 第一个迭代必须做 | 🟢 逐步完善

---

## 一、高可用 & 容灾（5 项）

### 1.1 中间件集群化

**差距**：docker-compose 中 MySQL/Redis/RabbitMQ/MongoDB/ES 全单节点。任何一个挂了整个系统不可用。

**后果**：Redis 宕机 → 所有服务缓存穿透打到 MySQL → MySQL 也挂 → 全站 502。

**怎么改**：
| 中间件 | 方案 | 具体操作 |
|--------|------|----------|
| MySQL | 主从 + MHA/Orchestrator | 1主2从，主库写、从库读。ShardingSphere-Proxy 配置读写分离规则。MHA 监控主库存活自动故障切换 |
| Redis | Sentinel 哨兵 | 1主2从 + 3 Sentinel 节点。`spring.redis.sentinel.nodes` 配置替代直连。自动故障转移 + 通知 |
| RabbitMQ | 镜像队列集群 | 3 节点集群，`ha-mode: all` 镜像所有队列。消息不会因为单节点挂了丢失 |
| MongoDB | Replica Set | 3 节点副本集，`mongodb://host1,host2,host3` 连接串。自动选主 |
| Elasticsearch | 3 节点集群 | 3 个 master-eligible 节点，`discovery.seed_hosts` 配置。索引分片 + 副本自动分布 |

**优先级**：🔴 上线前必须做

---

### 1.2 服务无状态化 + 多实例

**差距**：每个微服务单实例运行，JVM 崩溃 = 该服务不可用。

**后果**：Order 服务宕机 → 所有下单请求失败。没有自动重启、没有流量转移。

**怎么改**：
- 每个服务至少 2 个实例（K8s Deployment `replicas: 2`）
- 确保服务完全无状态（会话信息在 Redis，不在 JVM 内存）
- K8s Liveness Probe：`/actuator/health/liveness`，失败自动重启
- K8s Readiness Probe：`/actuator/health/readiness`，失败自动摘流量
- K8s `podAntiAffinity` 确保 2 个副本不在同一台物理机上

**优先级**：🔴 上线前必须做

---

### 1.3 熔断降级

**差距**：没有 Sentinel/Hystrix/Resilience4j。下游服务慢了就一直等，拖死上游。

**后果**：Payment 服务调支付渠道超时 30 秒 → 所有 Tomcat 线程被占满 → Order 调 Payment 也超时 → 级联雪崩。

**怎么改**：
- 引入 **Sentinel**（Spring Cloud Alibaba 生态，和 Nacos 天然集成）
- 对每个 Feign 调用配置：
  - **RT 熔断**：响应时间 > 1s 且比例 > 50%，熔断 10 秒
  - **异常比例熔断**：异常比例 > 50%，熔断
  - **降级逻辑**：支付查询降级 → 返回"支付处理中，请稍后刷新"；商品详情降级 → 返回 Redis 缓存
- Sentinel Dashboard 可视化查看熔断状态、实时 QPS

**优先级**：🔴 上线前必须做

---

### 1.4 网关限流

**差距**：抢券场景只有 Redis SET NX 互斥（应用层），Gateway 层无限流。脚本 10000 QPS 能直接打到下游。

**后果**：恶意刷接口导致服务 OOM，正常用户也无法使用。

**怎么改**：
- Gateway 层接入 **Sentinel Gateway 限流**：
  - `/api/v1/coupon/grab`：令牌桶，每秒 100 个请求
  - `/api/v1/auth/login/sms`：每分钟同一 IP 最多 3 次
  - `/api/v1/order/create`：每用户每秒 10 个
- 超出限流返回 `429 Too Many Requests`
- 限流规则放 Nacos 配置中心，动态生效不用重启

**优先级**：🔴 上线前必须做

---

### 1.5 异地容灾 & 数据备份

**差距**：单机房部署。机房断电/光纤挖断 → 全站不可用。数据库没有自动备份策略。

**后果**：MySQL 磁盘损坏 → 所有数据永久丢失。

**怎么改**：
- **数据库备份**：`mysqldump` 每日全量 + binlog 增量，保留 30 天。或直接用阿里云 RDS 自动备份
- **Redis 持久化**：`appendonly yes` + `aof-use-rdb-preamble yes`（已配置 RDB + AOF 混合持久化）
- **MongoDB 备份**：`mongodump` 每日全量
- **异地备份**：备份文件自动上传 OSS（跨 Region），防止同机房灾难
- **多活**（长期目标，成本高，可先不做）：2 个机房各部署一套，DNS 智能解析分流

**优先级**：🟡 第一个迭代必须做（备份）；异地多活 🟢 逐步完善

## 二、安全深度（4 项）

### 2.1 敏感数据加密存储

**差距**：用户手机号、身份证号等敏感信息可能明文存在 MySQL。

**后果**：数据库被拖库 → 几十万用户隐私泄露 → 合规处罚 + 公关危机。

**怎么改**：
- 手机号：AES-256 加密存储，密钥放 KMS（阿里云 KMS 或 HashiCorp Vault）
- 身份证号、银行卡号：同理，且脱敏展示（`138****1234`）
- MyBatis-Plus TypeHandler 自动加解密：写一个 `EncryptedStringTypeHandler`，insert 时自动加密、select 时自动解密
- 日志脱敏：logback 配置 `%replace` 模式，自动把手机号替换为 `138****1234`

**优先级**：🔴 上线前必须做

---

### 2.2 操作审计日志

**差距**：管理员删了 100 张券、修改了订单价格、手动退款了一笔——没有任何记录。

**后果**：内部人员恶意操作不可追溯、合规审查无法通过。

**怎么改**：
- 创建 `t_audit_log` 表（或直接存 ES）：
  ```
  id, tenant_id, user_id, user_name, action（CREATE/UPDATE/DELETE）,
  resource_type（ORDER/COUPON/SETTLEMENT）, resource_id,
  before_value（JSON）, after_value（JSON）,
  ip, user_agent, create_time
  ```
- 用 AOP 切面自动采集：`@AuditLog(action="DELETE", resourceType="COUPON")`
- 审计日志只追加不修改不删除（append-only），防止篡改
- 管理员后台提供审计日志查询界面（按时间、操作人、操作类型筛选）

**优先级**：🟡 第一个迭代必须做

---

### 2.3 渗透测试 & 安全扫描

**差距**：代码没有经过任何安全扫描。可能存在 XSS、CSRF、越权、SQL 注入等未知漏洞。

**后果**：上线后被白帽子/黑产发现漏洞并利用。

**怎么改**：
- **SAST（静态代码扫描）**：SonarQube 扫描所有 Java 代码，CI 流水线集成
- **DAST（动态扫描）**：OWASP ZAP 对运行中的系统做自动渗透测试
- **依赖漏洞扫描**：`mvn dependency-check:check` 检查第三方 JAR 已知 CVE
- **人工渗透**（重要）：找安全工程师手动测试：
  - 越权：学生 A 能看到学生 B 的订单吗？
  - 支付金额篡改：修改请求里的 amount 参数会怎样？
  - Token 伪造：伪造 JWT 能绕过鉴权吗？
- 修复所有高危/中危漏洞后再上线

**优先级**：🔴 上线前必须做（至少做完扫描 + 高危修复）

---

### 2.4 WAF & DDoS 防护

**差距**：Gateway 直接暴露在公网，没有 Web 应用防火墙。

**后果**：SQL 注入绕过代码层、CC 攻击打满带宽。

**怎么改**：
- **云 WAF**：阿里云 WAF / Cloudflare，部署在 Gateway 前面
  - SQL 注入、XSS、命令注入 → WAF 拦截
  - CC 攻击 → WAF 频率限制 + 验证码
  - 恶意 IP 库 → 自动封禁
- **CDN**：静态资源（前端页面、图片）走 CDN，减少源站压力
- Gateway 只允许 CDN/WAF 的回源 IP 访问，拒绝其他来源

**优先级**：🟡 第一个迭代必须做（云 WAF 开箱即用，几小时配好）

## 三、可观测性深度（3 项）

### 3.1 分布式链路追踪

**差距**：一个请求经过 Gateway → Order → MQ → Payment → Settlement → Notify，出问题只能逐个服务翻日志。

**后果**：线上故障定位时间以小时计。某个用户说"我付了款但订单还是待支付"，你不知道卡在哪一步。

**怎么改**：
- 接入 **SkyWalking**（推荐，Apache 顶级项目，Java Agent 无侵入）：
  - 每个服务 JVM 启动加 `-javaagent:skywalking-agent.jar`
  - SkyWalking OAP Server 收集数据、ES 存储
  - UI 界面查看完整调用链：每个 Span 的耗时、SQL、异常
- MQ 链路打通：生产者发消息时注入 traceId，消费者取出后继续链路
- `logback.xml` 加 `%traceId`，日志和链路追踪通过 traceId 串联

**优先级**：🟡 第一个迭代必须做

---

### 3.2 告警体系

**差距**：Prometheus 有指标但没人盯着 Dashboard。凌晨出问题没人知道。

**后果**：支付通道挂了 6 小时才被发现，损失几百单。

**怎么改**：
- **Prometheus AlertManager** 配置告警规则：

| 告警 | 条件 | 级别 | 通知方式 |
|------|------|------|----------|
| 服务宕机 | `up == 0` 持续 1 分钟 | P0 | 电话 + 钉钉/企微 |
| 支付成功率 | `pay_success_rate < 95%` 持续 5 分钟 | P0 | 电话 |
| 接口 P99 > 3s | `http_server_requests_seconds{quantile="0.99"} > 3` 持续 5 分钟 | P1 | 钉钉/企微 |
| JVM 堆内存 > 85% | `jvm_memory_used_bytes > 85%` 持续 10 分钟 | P1 | 钉钉 |
| 订单量归零 | `order_create_total[5m] == 0` | P1 | 钉钉 |
| 磁盘 > 85% | `node_filesystem_avail_bytes < 15%` | P2 | 钉钉 |

- **告警通知链**：钉钉群 → 5 分钟无人确认 → 电话 on-call
- **告警静默**：发布期间自动静默 15 分钟（CI/CD 流程里调 AlertManager API）

**优先级**：🟡 第一个迭代必须做

---

### 3.3 业务监控大盘

**差距**：没有面向运营/产品的实时数据看板。

**后果**：运营不知道今天 GMV 多少、转化率多少、哪个学校订单多。每次要数据找开发跑 SQL。

**怎么改**：
- **Grafana 业务大盘**（不和 JVM 大盘混在一起）：

| 面板 | 指标 | 数据来源 |
|------|------|----------|
| 实时 GMV | `sum(pay_amount_total[1h])` | Prometheus Counter（代码里埋点） |
| 订单量曲线 | `rate(order_create_total[5m])` × 时间 | Prometheus |
| 支付转化率 | `pay_success / order_create * 100%` | Prometheus |
| 券核销率 | `coupon_used / coupon_sent * 100%` | Prometheus |
| TOP10 学校 | 按 tenant_id 分组排序 | Prometheus |
| 服务健康一览 | `up` 指标 | Prometheus |

- **代码埋点**：用 Micrometer 的 `Counter`/`Timer` API 在关键业务节点埋点
  ```java
  // 下单成功
  meterRegistry.counter("order_create_total", "tenant_id", tenantId).increment();
  // 支付成功
  meterRegistry.counter("pay_success_total", "tenant_id", tenantId).increment();
  ```

**优先级**：🟢 逐步完善（先做核心指标：GMV、订单量、成功率）

## 四、数据治理（3 项）

### 4.1 冷热数据分离 & 归档

**差距**：订单表（`t_order_0` ~ `t_order_15`）无限增长。一年后单表千万级，`SELECT * FROM t_order_0 WHERE create_time < '2025-01-01' AND status = 'COMPLETED'` 全表扫描。

**后果**：MySQL 磁盘打满，历史查询越来越慢拖垮主库。

**怎么改**：
- **方案：按月分表 + 归档**
  - ShardingSphere 增加 `create_time` 作为分片条件：当月数据查热表，历史数据查归档表
  - 或简单方案：`t_order_YYYYMM` 按月建表，每月 1 号自动创建下月表
- **归档策略**：
  - 3 个月前的已完成/已取消订单 → ETL 到 ClickHouse/ Doris（OLAP）
  - OLAP 里做统计分析（GMV 趋势、品类排行）
  - MySQL 保留 6 个月数据，更早的物理删除（OLAP 已有备份）
- **定时任务**：XXL-JOB 每月执行归档 + 清理

**优先级**：🟢 逐步完善（上线 3 个月后数据量才会成为问题）

---

### 4.2 数据一致性校验

**差距**：MySQL → Canal → MQ → ES 这条链路，ES 数据可能漏同步或延迟过大。

**后果**：用户在搜索框搜到已下架的商品、或者上架的商品搜不到。

**怎么改**：
- **定时全量对账**：XXL-JOB 每小时跑一次
  - 从 MySQL 查最近 1 小时变更的 SPU 列表
  - 逐一在 ES 中 `GET /campus_product_spu/_doc/{spuId}`
  - 不一致的自动修复（重新索引）
- **对账结果监控**：不一致数量 > 阈值 → 告警
- **兜底修复**：对账脚本暴露 HTTP 接口，可手动触发某个 SPU 的全量重索引

**优先级**：🟡 第一个迭代必须做（Canal 刚上线时最容易出问题）

---

### 4.3 数据脱敏（非生产环境）

**差距**：开发/测试环境可能导入了生产数据，用户隐私直接暴露给开发人员。

**后果**：开发人员误看/误传用户数据 → 隐私泄露。

**怎么改**：
- **生产数据 → 测试环境**必须经过脱敏脚本：
  - 手机号：保留前 3 后 4，中间替换为 `****` → `138****1234`
  - 身份证：全部替换为随机合法格式的假号
  - 姓名：随机替换为假名
  - 地址：替换为城市 + 随机街道
- 脱敏脚本写好后固化到 CI/CD，禁止未脱敏数据进入非生产环境

**优先级**：🟡 第一个迭代必须做

## 五、发布 & 运维（3 项）

### 5.1 灰度发布

**差距**：新版本上线是 15 个服务全量替换，Bug 影响所有用户。

**后果**：一个 Order 服务的空指针，导致所有学校都不能下单。

**怎么改**：
- **方案：Spring Cloud Alibaba 全链路灰度**
  - Nacos 给灰度实例打标签 `version: canary`、`version: stable`
  - 请求头带 `X-Version: canary` → 流量只走灰度实例
  - 5% 流量先进金丝雀，观察 30 分钟无异常 → 全量发布
- **轻量方案**（如果不想引入全链路灰度）：
  - K8s 中先起 1 个新版本 Pod（`replicas: 1`），跑 10 分钟
  - 看日志/监控无异常 → `kubectl scale --replicas=N` 扩到目标数量
  - 逐个缩容旧版本 Pod

**优先级**：🟢 逐步完善（先做 K8s 分批发布，再做全链路灰度）

---

### 5.2 压力测试

**差距**：不知道系统能扛多少 QPS。注册、下单、抢券的瓶颈在哪，扩容到几台机器能撑住。

**后果**：开学季流量翻 10 倍，系统直接崩，错过全年最大的一波用户。

**怎么改**：
- **工具**：JMeter / Locust / K6
- **测试场景**：

| 场景 | 并发数 | 目标 QPS | 关键指标 |
|------|--------|----------|----------|
| 商品列表查询 | 500 并发 | 5000 QPS | P99 < 200ms |
| 用户注册 | 200 并发 | 1000 QPS | 成功率 ≥ 99.9% |
| 下单 | 300 并发 | 1500 QPS | P99 < 500ms |
| 抢券 | 1000 并发 | 5000 QPS | 不超发、不崩溃 |
| 混合场景 | 500 并发 | — | 模拟真实流量比例 |

- **每轮压测后**：
  - 看 Grafana 找瓶颈（CPU？内存？数据库连接池？Redis？）
  - 优化瓶颈 → 重新压测
- **最终产出**：《压测报告》——每个接口的极限 QPS、推荐机器配置、扩容方案

**优先级**：🟡 第一个迭代必须做

---

### 5.3 故障演练

**差距**：不知道 MySQL 挂了系统会怎样，也不知道怎么恢复。

**后果**：第一次遇到 MySQL 宕机时手忙脚乱，恢复时间 2 小时。

**怎么改**：
- **混沌工程**：用 ChaosBlade 定期演练

| 演练场景 | 注入方式 | 验证点 |
|----------|----------|--------|
| MySQL 宕机 | 关闭 MySQL 容器 | Sentinel 熔断是否生效？业务是否降级？恢复后是否自动重连？ |
| Redis 宕机 | 关闭 Redis 容器 | 缓存穿透是否被熔断拦截？恢复后数据是否完整？ |
| 某个服务宕机 | kill Pod | K8s 是否自动重启？Gateway 是否摘掉不健康实例？ |
| 网络延迟 | `tc qdisc` 注入 500ms 延迟 | 调用链超时配置是否合理？是否级联雪崩？ |
| 磁盘满 | `dd` 写满磁盘 | 日志轮转是否生效？告警是否触发？ |

- **产出**：《故障演练报告》+ 《应急预案》——每种故障的恢复步骤
- 频率：每月一次

**优先级**：🟢 逐步完善（上线后再做，初期靠真实故障积累经验）

## 六、多租户 SaaS 深度（3 项）

### 6.1 租户计费 & 订阅管理

**差距**：每个学校怎么收费？免费的和付费的有什么区别？完全没有计费模块。

**后果**：商业模式不成立——不知道怎么向学校收钱。

**怎么改**：
- 新建 **Billing 微服务** 或扩展现有 Tenant 服务：

| 表 | 作用 |
|------|------|
| `tenant_package` | 套餐定义（免费版/标准版/企业版），月费、订单上限、存储上限 |
| `tenant_subscription` | 租户订阅记录，套餐ID、有效期、状态 |
| `tenant_usage` | 租户用量（本月订单数、存储量、用户数） |
| `billing_invoice` | 账单/发票 |

- **计费策略**：
  - 免费版：限 50 单/月，1GB 存储，1 所学校
  - 标准版：¥999/月，1000 单/月，50GB 存储
  - 企业版：¥3999/月，无限订单，500GB 存储，专属客服
- **超额处理**：用量达到上限 → 短信/邮件提醒管理员升级 → 超限后限制下单（返回提示"本月订单已达上限请升级套餐"）

**优先级**：🟡 第一个迭代必须做（否则无法收费）

---

### 6.2 租户资源隔离

**差距**：学校 A 的抢券活动把 Redis 打满了，学校 B/C/D 跟着一起慢。

**后果**：一个租户的不良行为影响所有人。

**怎么改**：
- **Redis Key 前缀租户隔离**：所有 Key 加 `{tenantId}:` 前缀（代码里已部分支持，需要全量确认）
- **数据库连接池独立**：大租户单独配置连接池上限，防止占满所有连接
- **Sentinel 租户级限流**：每个租户独立限流阈值——企业版比免费版高
- **消息队列租户隔离**：RabbitMQ 为每个租户创建独立 vhost（成本高）→ 或在消费端按 tenantId 做优先级排队

**优先级**：🟢 逐步完善（先做 Key 前缀 + 限流隔离）

---

### 6.3 租户自助入驻 + 管理后台

**差距**：新学校接入需要开发人员手动改数据库创建租户。没有自助注册流程。

**后果**：规模化扩展时，每接入一个学校都要人肉操作，交付效率极低。

**怎么改**：
- **自助入驻流程**：
  1. 学校管理员访问注册页 → 填写学校名称、联系方式
  2. 系统自动创建 Tenant 记录 + 初始化数据库 schema + 默认角色和权限
  3. 发送激活邮件（含管理后台地址）
  4. 学校管理员登录管理后台 → 导入师生数据 → 配置服务类目 → 正式使用
- **管理后台功能**（按角色区分）：

| 角色 | 功能 |
|------|------|
| 学校管理员 | 师生管理、服务类目配置、价格设置、订单统计、师傅管理 |
| 师傅 | 接单/拒单、我的收入、提现 |
| 学生 | 浏览服务、下单、支付、评价、我的订单 |
| 运营（平台方） | 学校管理、套餐管理、数据大盘、问题处理 |
| 超管（平台方） | 所有权限 + 系统配置 + 审计日志 |

- **前端**：Vue3 + Element Plus 或 React + Ant Design（两个独立前端工程：学生端 app + 管理后台 admin）

**优先级**：🟡 第一个迭代必须做（最小可用：学校入驻 + 管理员后台）
