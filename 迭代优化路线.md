## Phase 1：补基础 + 堵漏洞（2 周）

> 目标：全链路跑通 + 低成本高风险漏洞清零

### 1. 全链路联调测试

**现状**：15 个服务各自独立 curl 通过，但从未串联测试。

**联调场景**：

```
场景 A — 核心交易链路：
  学生注册(Auth) → 浏览商品(Product) → 下单(Order) → 支付(Payment)
    → 订单状态流转(Order) → 师傅钱包入账(Settlement)
    → 评价(Review) → 消息推送(Notify)

场景 B — 营销券链路：
  运营创建优惠券(Coupon) → 学生抢券(Coupon) → 下单时使用(Order)
    → 券核销(Coupon) → 结算时扣券(Settlement)

场景 C — IM 通信链路（含 WebSocket Gateway 路由验证）：
  下单 → 自动创建会话 → 学生发消息 → 师傅收到
    → 离线消息拉取 → 系统消息推送
```

**验证点**：Feign 调用超时、MQ 消息丢失、状态机流转字段完整性。

**联调中预期会暴露的 Bug（需顺手修复）**：
- 🔴 `PaymentService.handleCallback()` NPE 隐患 —— `existing` 为 null 时崩溃
- 🟡 Gateway 缺少 WebSocket 路由（`/ws` 只能直连 8090，需加 WebSocket 转发规则）
- 🟡 `OrderService.myOrders()` 分页实际全量返回（依赖 1-2 分页插件恢复）

### 2. 修复 MyBatis-Plus 分页插件

**现状**：`MybatisPlusConfig` 类级别 `// @Configuration` 和 @Bean 全部注释。`OrderService.myOrders()` 调了 `selectPage()` 但因分页插件未注册，实际返回全量数据（假分页）。

**根因回顾**：当初 StackOverflow 的根因是 UserContext 的 Lombok `@Data`，不是 Security 与 MP 的循环依赖。UserContext 已在调试阶段去掉 Lombok 并手写 getter/setter。

**方案**：直接取消注释 `@Configuration` 和 `@Bean`，大概率即开即用。测试 `selectPage` 是否正常分页。

**⚠️ 可行性验证**：已验证 UserContext.java 不再使用 Lombok，原 StackOverflow 根因已消除。直接恢复即可，不需要额外排查循环依赖。

### 3. 租户拦截器恢复 + 多租户隔离测试

**现状**：`TenantInterceptor` 已定义但从未注册为 Bean（`MybatisPlusConfig` 被整类注释）。SQL 中无 `WHERE tenant_id = ?`，所有租户数据混在一起。

**⚠️ 代码验证发现的隐患**：`Order` 实体不继承 `BaseEntity`（它自己定义了 createTime/updateTime，没有 isDeleted 字段）。恢复拦截器后，Order 的 insert 时 tenant_id 可能不会被 MP 的 MetaObjectHandler 自动填充——需要确认哪些 Entity 继承了 BaseEntity，哪些没有。

**方案**：
1. 恢复 MybatisPlusConfig 的同时注册 TenantInterceptor
2. 全量排查所有 Entity 是否继承 BaseEntity，未继承的统一改造
3. 验证多租户场景下数据隔离正确性

### 4. Security 放行改回 Gateway 鉴权模式 + WebSocket 路由

**现状**：所有下游服务 `SecurityConfig` 放行全部请求（`.anyRequest().permitAll()`）。Gateway 挂了任何服务裸奔。此外 Gateway 缺少 WebSocket 路由规则，`/ws` 端点只能直连 IM 服务 8090 端口。

**方案**：
- Gateway 保持 JWT 鉴权 + 白名单
- 下游服务改为仅信任来自 Gateway 的请求（验证共享 Secret Header 或内网 IP）
- 直接访问 `http://localhost:8082/api/xxx` 无 Token 时返回 401
- Gateway 新增 WebSocket 路由：`/ws/**` → `lb://campus-im`（或 `http://localhost:8090`）

### 5. Gateway 切回 `lb://` 路由

**现状**：Gateway 路由全部用 `http://localhost:808X` 直连，绕过 Nacos 注册中心。

**前置条件**：Nacos 服务注册正常（所有服务 `spring.cloud.nacos.discovery` 已配置但未使用）。

**方案**：恢复 `lb://campus-user` 等前缀，解决 Header 透传问题后切换。

### 6. Redis 设密码 + CORS 收紧白名单

**现状**：
- Redis 无密码（docker-compose 未设置 `requirepass`，所有服务 `spring.data.redis.password` 为空）
- Gateway CORS 配置 `allowedOriginPatterns: "*"`

**⚠️ 可行性提醒**：Redis 加密码需要同步修改 15 个服务的 `application.yml`（加 `spring.data.redis.password`），漏一个服务就会导致该服务所有 Redis 操作报 NOAUTH。建议先启用 Nacos 配置中心（Task 1-5），将 Redis 密码统一放在 Nacos 中管理。

**方案**：
- Docker Redis 设置 `requirepass`
- 通过 Nacos 配置中心统一下发 `spring.data.redis.password`（需 Task 1-5 先完成）
- CORS `allowedOriginPatterns` 改为具体前端域名列表

### 7. IM MongoDB 加 `sessionId` 索引

**现状**：`ImMessage` 集合按 `sessionId` 查询消息历史，无索引。

**方案**：`db.im_message.createIndex({sessionId: 1, createTime: -1})`

### 8. Order Service 防 `updateById` 误用

**现状**：`orderMapper.updateById(order)` 会触发 ShardingSphere `Can not update sharding value` 拒绝。已在 Service 层全部替换为 `LambdaUpdateWrapper` 精准更新，但无法防止后续新人误用。

**❌ 原方案（AOP）不可行**：Spring AOP 无法拦截 MyBatis-Plus 的 `BaseMapper.updateById()`——MP 的 Mapper 是接口 + JDK 动态代理，AOP 切面只能拦截 Spring Bean 的方法，Mapper 代理对象不在此范围内。

**修正方案**（三选一）：
- **方案 A（推荐）**：在 `MybatisPlusInterceptor` 中添加自定义 `InnerInterceptor`，拦截 UPDATE SQL，检测到 `t_order` 表且 SET 子句包含分片键（`order_id`/`tenant_id`）时抛异常
- **方案 B**：在 `OrderMapper` 接口中用 `default` 方法覆盖 `updateById`，标记 `@Deprecated` 并抛 `UnsupportedOperationException`
- **方案 C**：不写代码，在 OrderService 类上打`// 禁止使用 updateById，必须用 LambdaUpdateWrapper`注释 + Code Review 规范

### 9. `@TableLogic` 逻辑删除 + Entity 继承关系全量排查

**现状**：`BaseEntity` 有 `@TableLogic` 注解，但之前排查 StackOverflow 时注释过 `MybatisPlusConfig`（包含分页插件）。当前 `@TableLogic` 注解本身未被注释，但由于 MybatisPlusConfig 被整类注释，逻辑删除功能是否生效取决于 MP 自动配置。

**⚠️ 代码验证发现**：`Order` 实体不继承 `BaseEntity`（自己定义了 createTime/updateTime，没有 isDeleted 字段）。这意味着即使恢复逻辑删除，对 Order 表仍然是物理删除。

**方案**：
1. 逐服务确认哪些 Entity 继承了 BaseEntity，哪些没有
2. 未继承的统一改造（继承 BaseEntity 或至少加 `@TableLogic` 字段）
3. 恢复 MybatisPlusConfig 后，用 SQL 验证 `DELETE` 实际执行为 `UPDATE SET is_deleted=1`

### 10. 🆕 接入定时任务框架（XXL-JOB 或 @Scheduled）

**⚠️ 代码验证发现**：全项目 0 个 `@Scheduled` 注解——订单超时取消、结算对账、优惠券过期清理等核心定时任务**完全缺失**。之前路线图写"XXL-JOB 替换 @Scheduled"的前提不成立，不是"替换"而是"从零创建"。

**影响**：
- 待支付订单永远不会自动超时取消
- 结算对账无自动执行机制
- 过期优惠券状态不会自动更新

**方案**：
1. 梳理所有需要定时任务的业务场景（订单超时取消、对账、券过期、数据清理）
2. 先用 Spring `@Scheduled` 快速实现（Phase 1），确保业务逻辑正确
3. 后续升级为 XXL-JOB（Phase 3），获得调度面板、失败重试、分片能力

## Phase 2：提质量 + 补功能（2 周）

> 目标：核心 Service 有测试 + 关键功能 TODO 补完 + 文档聚合

### 11. Auth / Order / Coupon 核心 Service 单元测试

**现状**：项目零测试文件、零测试依赖。所有验证靠手动 curl。Order 测试尤其复杂——依赖 ShardingSphere-Proxy 分库分表，H2 无法模拟。

**方案**：
- 根 POM 加 `spring-boot-starter-test`
- 先从 Auth 开始（不依赖分库分表），再覆盖 Coupon、Order
- Order 用 Testcontainers + MySQL 模拟分库分表环境，或直接连 Proxy 的测试库
- 测试范围：注册/登录/Token 刷新、创建订单/状态流转/取消、抢券并发/批量发券/核销

### 12. JWT 密钥改 Nacos 配置 + Jasypt 加密敏感配置

**现状**：
- `JwtTokenProvider.DEFAULT_SECRET` = `"ShareCampusCloud2025SecretKeyForJWT!!!!"` 硬编码明文
- 数据库密码 `root/root123` 明文写在 `application.yml`
- `Nacos config` 在所有 bootstrap.yml 中 `enabled: false`

**⚠️ 前置条件**：Nacos Config 必须先启用（依赖 Task 1-5 lb:// 路由恢复中的 Nacos 环境就绪）。

**方案**：
- 先启用 Nacos Config
- JWT Secret 移到 Nacos 配置中心，支持动态刷新
- 所有密码类配置用 Jasypt 加密（`ENC(...)` 格式），解密密钥通过环境变量传入

### 13. 订单快照补全

**现状**：`OrderSnapshot` 实体已定义（`@TableName("t_order_snapshot")`），`saveSnapshot()` 方法逻辑已写好但 `snapshotMapper.insert(snapshot)` 被注释，`getSnapshot()` 返回 `null` + `// TODO`。

**方案**：
1. 恢复 snapshotMapper.insert() 调用
2. 实现 getSnapshot() 从 t_order_snapshot 表查询
3. 验证 detail() 接口返回完整订单 + 快照数据

### 14. 支付回调验签真实实现

**现状**：`MockPaymentChannel.verifyCallback()` 永远 `return true`。

**⚠️ 注意**：Mock 实现是开发环境的设计选择，不是 bug。真实对接微信/支付宝需要商户资质。

**方案**：
- 保留 `MockPaymentChannel`（`@Profile("dev")`）
- 新增 `WxPaymentChannel` 实现 `PaymentChannel` 接口（`@Profile("prod")`）
- 实现真实 RSA/SHA256 验签逻辑

### 15. OSS 前端直传签名签发

**现状**：File 服务只做了服务端接收（`MultipartFile`），前端直传 OSS 的 STS 临时凭证签发未实现。

**方案**：File 服务新增 `/api/v1/file/sts-token` 接口，利用 MinIO `getPresignedObjectUrl()` 返回临时上传凭证。前端拿到后直传 MinIO，减少服务端带宽压力。

### 16. Knife4j Gateway 聚合文档

**现状**：15 个服务均有 `knife4j-openapi3-jakarta-spring-boot-starter`，但 Gateway 缺少聚合组件。

**方案**：Gateway 加 `knife4j-gateway-spring-boot-starter`，配置聚合路由将各服务 OpenAPI 文档统一在 `/doc.html` 呈现。

### 17. 架构说明书更新

**现状**：架构书中写的 ShardingSphere JDBC 内嵌、Seata AT 模式、Redisson，与实际不符。

**方案**：更新以下章节：
- ShardingSphere JDBC → Proxy 架构变更及原因（SnakeYAML 冲突）
- Seata → MQ+定时对账最终一致（决策理由）
- Redisson → StringRedisTemplate（Windows Docker 兼容性问题）
- 新增：架构设想 vs 实际差异对照表（来自本文档开头）

### 18. 🆕 服务健康检查（Actuator 接入）

**⚠️ 代码验证发现**：所有服务 POM 中零 actuator 依赖。Gateway 无法判断下游服务是否存活，负载均衡器没有健康检查端点。

**方案**：
- 所有服务加 `spring-boot-starter-actuator`
- 暴露 `/actuator/health` 端点（供 Gateway/负载均衡器使用）
- 后续 Phase 3 升级为 Prometheus 指标暴露

## Phase 3：加中间件能力（2 周）

> 目标：中间件从"跳过"到"真正跑通"

### 19. Canal → ES 数据同步

**现状**：SearchService 的 MQ 消费者 `handleProductSync()` 已写好，MQ 队列 `DB_SYNC_PRODUCT_QUEUE` 已定义。Canal Server 在 docker-compose 中已有完整配置（注释状态）。只缺 Canal 启动和 instance.properties 配置。

**⚠️ 可行性评估**：基础设施就绪程度高于预期——MQ 消费端代码完成，docker-compose 已有模板。主要工作量在 Canal 的 binlog 监听配置和测试验证。

**方案**：
- 取消 docker-compose 中 Canal 注释并启动
- 配置 `instance.properties`（MySQL 连接、binlog 监听表、MQ 投递目标）
- 验证：MySQL INSERT → Canal → MQ → SearchService.handleProductSync() → ES 索引
- 商品 SPU/SKU 变更实时同步到 ES

### 20. XXL-JOB 替代 @Scheduled（升级定时任务）

**现状**：Phase 1-10 已用 `@Scheduled` 实现了基础定时任务。现在升级为 XXL-JOB 以获得企业级调度能力。

**方案**：
- 部署 XXL-JOB Admin
- 所有 `@Scheduled` 方法迁移为 XXL-JOB Handler
- 获得：可视化调度、失败重试、分片并行、执行日志

### 21. Redisson 排查 + WatchDog 自动续期

**现状**：`CouponService.sendOne()` 和 `PaymentService.handleCallback()` 用 `StringRedisTemplate.setIfAbsent(key, "1", Duration.ofSeconds(10))`。功能等效但锁过期时间固定 10 秒，长事务场景下可能在业务完成前释放。

**排查方向**：
- 在 Linux 环境（非 Windows Docker）测试 Redisson 连接
- 升级 Redisson 版本 → 最新版
- 仍不行则评估 Lettuce 方案或手动实现 WatchDog 续期

### 22. Seata 接入 或 MQ+对账最终一致方案定稿

**现状**：分布式事务完全依赖 MQ + 定时对账，无强一致性保障。

**方案**（二选一）：
- **方案 A**：接入 Seata AT 模式 —— 成本高，需要 Seata Server + undo_log 表
- **方案 B**：明确走最终一致 —— MQ + 本地消息表 + 定时对账 + 人工介入后台

**建议**：校享云场景（下单→支付→结算）对实时一致性要求不高，方案 B 更务实。

### 23. Prometheus + Grafana 监控接入

**现状**：零监控。所有服务缺少 actuator + micrometer 依赖。

**⚠️ 前置条件**：Task 2-18 已接入 Actuator。

**方案**：
- 加 `micrometer-registry-prometheus` 依赖
- 暴露 `/actuator/prometheus` 端点
- Grafana Dashboard 可视化 JVM 指标（堆内存、GC、线程数）

## Phase 4：备生产（1 周）

> 目标：从"开发环境能跑"到"可以部署上线"

### 24. Docker Compose → K8s 部署

**⚠️ 现状**：项目零 Dockerfile。需为 15 个服务各写一个（可模板化生成）。

**方案**：
- 15 个服务编写 Dockerfile（多阶段构建，Alpine 基础镜像）

- Nacos / Gateway / ShardingSphere-Proxy 集群化配置

### 25. CI/CD Pipeline

**方案**：
- GitHub Actions 或 Jenkins
- PR → 自动编译 + 单元测试 → 构建镜像 → 推送镜像仓库
- Merge → 自动部署到开发/测试环境

### 26. ELK 日志收集

**现状**：日志散落在各服务 `logs/` 目录，排查问题需逐个翻文件。

**方案**：
- Filebeat 采集日志 → Elasticsearch → Kibana
- 或轻量方案：Loki + Grafana（省资源）

| 4 | 🟡 中 | `Order.java` | **不继承 BaseEntity**：缺少 `isDeleted`/`@TableLogic`，缺少 MP 自动填充（`createTime`/`updateTime`/`tenantId`） | DELETE Order → 物理删除；insert 时 tenant_id 需手动 set |
| 5 | 🟢 低 | `CouponService.java:99-101` | **锁无续期**：`setIfAbsent(key, "1", Duration.ofSeconds(10))` 固定 10 秒过期，JVM 崩溃或 GC 暂停 >10s 时锁提前释放 | 极端情况（Full GC 时锁过期） |
| 6 | 🟢 低 | 全局 | **零 Actuator**：所有服务无健康检查端点，Gateway/K8s 无法判断服务存活 | 服务假死时流量仍被路由 |
